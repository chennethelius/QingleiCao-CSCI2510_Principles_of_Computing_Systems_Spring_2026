   can read-modify-write the shared variable, so no updates get lost.
   The adder performs exactly +20,000,000 and the subtractor performs

   This is ~7x longer than the racy version.
   The extra time is the cost of 40,000,000 lock/unlock pairs plus
   the two threads fighting for the same lock.

7.
   race = 0
   ./mutex_per_thread  0.04s user 0.00s system 24% cpu 0.165 total
   race = 0
   ./mutex_per_thread  0.03s user 0.00s system 98% cpu 0.033 total
   race = 0
   ./mutex_per_thread  0.03s user 0.00s system 98% cpu 0.031 total

   Average real time = (0.165 + 0.033 + 0.031) / 3 = 0.0763 s

   This is roughly the same as the racy program (~0.075 s). The
   mutex is acquired only twice total, so the locking overhead is
   negligible.

8. Per-iteration locks 40M times and the threads keep fighting for
   the lock, so they end up waiting on each other instead of running
   in parallel. Per-thread only locks twice, so one thread just runs
   its whole loop, then the other does. So basically same
   speed as the racy version.

9. Per-iteration is better when threads do lots of independent work
   and only briefly use shared state. Per-thread is better when the whole job needs to be prioritized,
   like writing a full log entry without it getting interleaved, or
   swapping in a new shared data structure all at once. Simpler and
   avoids tons of lock/unlock overhead.
